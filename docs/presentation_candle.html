📉 핵심 기능 흐름 - Trading Service

안정적 자산 예약 및 수수료 계산 주문 프로세스

💵 매수(BUY) 자산 예약 공식

주문 생성 시 즉시 총자산을 차감하면 체결 거부 및 취소 시 복구가 어렵습니다. 가용 금액에서 주문 원금 + 수수료(0.015%) 만큼 계산하여 예약(Reserved) 상태로 전환해 이중 지출을 원천 방지합니다.

🚫 대기 주문 단일화 룰

특정 유저가 특정 종목에 대해 미체결(Pending) 대기 주문을 가진 경우, 시장가 급변 시 중복 거래 체결 위험이 있으므로 새로운 대기 주문 진입을 도메인 유효성 검증 레벨에서 사전 기각합니다.

매수(BUY) 주문 비즈니스 계산 로직 거래 안전 원칙
1. 거래 원금 산출
체결 요청 수량(Quantity) × 목표 단가(Price) = 매수 원금 (Amount)
2. 거래 수수료 계산
매수 원금 × 수수료율 (0.00015) = 수수료액 (Fee) (원 단위 반올림 처리)
3. 가용 자산 예약 검증
- 필요 자산 합계: 매수 원금 + 수수료액
- [검증 조건]: 유저 계좌의 가용 예수금 < 필요 자산 합계 일 경우, 즉각 '잔고 부족 조건 불일치(FAILED_PRECONDITION)' 오류를 전사 반환하고 프로세스를 조기 종료.

🔥 분산 정합성 문제와 실태

신뢰할 수 없는 분산 환경에서의 데이터 손실 요인

👻 네트워크 일시 타임아웃 오류

BFF가 주문을 접수하고 마이크로서비스에 gRPC로 전달하여 DB 커밋은 성공했으나, 회신용 네트워크 선로가 불안정하여 타임아웃 오류가 발생하면 유저는 주문 실패로 오인하고 동일 주문을 연속 재시도하여 중복 주문이 체결되는 대형 사고가 발생합니다.

👻 RDBMS와 외부 카프카의 원자성 결여

주문 내역을 데이터베이스에 정상적으로 Insert 했으나, 카프카 브로커에 메시지를 전송하려는 순간 네트워크 끊김이나 장비 장애가 발생하여 이벤트 발행에 실패할 경우 주문은 체결되었지만 포트폴리오 갱신 및 미션 클리어가 누락되어 서비스 일관성이 파괴됩니다.

📦 해결책 2 - Transactional Outbox

로컬 트랜잭션을 활용하여 DB 쓰기와 이벤트 발행을 동기화

1. 로컬 트랜잭션 내 원자적 동시 저장

비즈니스 테이블(주문 원장 등)을 갱신하는 쿼리와, 외부로 전송할 카프카 발행 대기 이벤트 기록(Outbox Record)을 동일한 RDBMS 로컬 트랜잭션 @Transactional 범위 안에서 동시에 Insert 함으로써 둘의 저장 상태를 하나로 결합합니다.

2. Message Relay 프로세스의 비동기 발송

백그라운드에서 동작하는 메시지 릴레이 데몬이 Outbox 테이블을 주기적으로 스캔하여 미발행 상태의 메시지를 Kafka Broker로 안전하게 푸시한 뒤, 발행 완료 상태로 전환시킵니다.

⚡ Kafka 장애극복 최종 일관성 보장 시나리오
카프카 클러스터가 일시 다운(Down)되는 재난 상황이 발생하더라도, 마이크로서비스의 주문 로직은 로컬 트랜잭션(Local DB 쓰기)에 의해 중단 없이 정상 처리됩니다. 카프카 복구 시 메시지 릴레이가 데이터베이스에 안전하게 보관되어 있던 미전송 레코드를 찾아 순차적으로 재발행함으로써 시스템 간 최종 일관성(Eventually Consistency)을 확보합니다.

📂 서비스 DB 직접 접근 금지

Candle Batch는 타 마이크로서비스의 DB 테이블을 직접 조인·수정하지 않습니다. 예약·주문·잔고·종가·스냅샷·랭킹 데이터는 각 서비스가 소유하며, Batch는 공식 gRPC API를 통해 필요한 작업만 요청합니다. 도메인 상태 변경과 Outbox 기록은 소유 서비스의 트랜잭션에서 처리되고, Batch transaction은 원격 서비스 transaction을 대체하지 않습니다.

📂 Batch 전용 실행 경계 분리

시간 기반 대용량·일괄 작업(예약·주문 처리, Portfolio EOD, Ranking 확정, Stock Sync)은 API 서버가 아닌 Candle Batch에서 독립 실행합니다. Automatic Scheduler와 Batch Control gRPC 수동 요청은 모두 JobOperator로 합쳐져 단일 창구로 제어되며, Metadata는 PostgreSQL candle.batch 스키마에 격리 저장됩니다.

로컬 개발자 구동 및 테스트 워크플로우 Shell 명령어 구성
1. 환경 변수 초기화 복제
각 서비스별 개발 환경 템플릿 복사생성 (.env.development.example ──► .env)
2. 인프라 컨테이너 일괄 기동
도커 컴포즈 실행 명령 (docker-compose up -d) 실행을 통해 외부 저장소 격리 실행
3. 특정 서브모듈 독립 실행
Gradle 태스크 지정 실행 (./gradlew :services:trading-service:bootRun)

🗺️ 향후 개선 및 확장 계획

🔍 분산 Tracing 모니터링

OpenTelemetry 프로토콜 및 Zipkin 분산 트레이서 시각화 도구를 주입하여 BFF, 백엔드 서비스, 카프카 컨슈머 간을 횡단하는 트랜잭션의 지연 시간 병목 및 에러 원점 추적 자동화

⚖️ nGrinder 스트레스 튜닝

동시 사용자 접속 폭증 상황을 구현하는 nGrinder 부하 테스트망을 가동하여 가상 거래 매수/매도 지점을 검증하고, gRPC 커넥션 및 JVM ZGC(Garbage Collector) 튜닝 진행

🏦 실증 증권사 API 포팅

거래 체결 모의 가상 코어 계층의 추상 인터페이스를 확장하여 실제 한국투자증권/키움증권 오픈 API 드라이버를 결합함으로써 실계좌 연동 주문 처리가 가능한 유연한 설계 달성
